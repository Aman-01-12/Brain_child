# Kick / Remove + Ban Set — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prevent kicked participants from rejoining the same meeting by adding a Redis-backed ban set that is checked at both the REST join endpoint and the Socket.IO join handler.

**Architecture:** `/room/remove` already removes participants from LiveKit and Redis. We add a `room:{code}:banned` Redis SET (TTL = room TTL = 24h). On every join attempt, we check this set before anything else. The client-side change is only the alert text — all enforcement is server-side, so it's impossible to bypass without a new identity.

**Tech Stack:** Node.js/Express/Redis (ioredis), ObjC (AppDelegate.m), XCTest

  socket.on('client:join-room', async ({ roomCode, identity }) => {
    if (!roomCode || !identity) return;
    const code = String(roomCode).toUpperCase().replace(/[^A-Z0-9]/g, '');
    if (!code) return;

    // Ban check — prevent banned users from receiving room events even if

        socket.disconnect(true);
        return;
