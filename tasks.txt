## Architectural Decisions

These were resolved before the plan was written. They inform every step below.

### Decision 1: Audio — Single-Source Fan-Out [G1]

**Problem**: LiveKit's Swift SDK uses `AVAudioEngine` for mic capture. The app already uses `AVCaptureAudioDataOutput`. Running both on the same mic can cause conflicts on some hardware.

**Resolution**: Audio captured ONCE via existing `AVCaptureAudioDataOutput`. Raw `CMSampleBuffer`s fanned out to two consumers:
- `InterRecordingSink` (existing, disk)
- `InterLiveKitAudioBridge` (new, network — converts CMSampleBuffer → AVAudioPCMBuffer → LiveKit AudioCapturer)

LiveKit's built-in `LocalAudioTrack.createTrack()` is NOT used. Custom `AudioCapturer` replaces it.

### Decision 2: Camera — Add AVCaptureVideoDataOutput

**Problem**: `InterLocalMediaController` has no `AVCaptureVideoDataOutput`. LiveKit needs raw frames.

**Resolution**: Add `AVCaptureVideoDataOutput` to the existing `AVCaptureSession`. Zero-copy — CVPixelBuffer passed by reference. Do NOT use LiveKit's `CameraCapturer` (creates duplicate session).

### Decision 3: Swift / Objective-C Bridging

**Problem**: Codebase is Objective-C. LiveKit SDK is Swift.

**Resolution**: New networking files in Swift with `@objc` annotations. Bridging header imports ObjC headers. ObjC side imports `inter-Swift.h`.

### Decision 4: Entitlement Gap

**Problem**: Sandbox has no network entitlement. WebRTC needs TCP + UDP.

**Resolution**: Add `com.apple.security.network.client = YES`.

### Decision 5: Camera/Mic Toggle Ordering [G2]

**Problem**: `setCameraEnabled:NO` stops the capture device. LiveKit mute is a separate signal. Order matters to avoid black/frozen frames on remote side.

**Resolution**: Two-phase toggle with state machine (`active → muting → muted → enabling → active`):
- **Disable**: Mute LiveKit track FIRST → then stop capture device.
- **Enable**: Start capture device FIRST → wait for first real frame → then unmute LiveKit track.

### Decision 6: Subscriber Pixel Format [G3]

**Problem**: LiveKit/WebRTC decoder outputs NV12 (YUV). Can't render as BGRA.

**Resolution**: Dedicated Metal shader in `InterRemoteVideoView`. YUV→RGB via BT.709 matrix. Zero-copy via `CVMetalTextureCache`. Format detection supports NV12, full-range NV12, and BGRA fallback.

### Decision 7: Mode Transitions [G4]

**Problem**: Normal↔Interview mode switches destroy and recreate all media controllers. What happens to the LiveKit room?

**Resolution**: Room persists across mode transitions (owned by AppDelegate). Publishers are unpublished and rebuilt. Subscribers stay active. No reconnect delay. `transitionMode()` API on `InterRoomController`.

### Decision 8: Router Queue Bottleneck [G5]

**Problem**: `InterSurfaceShareController` routes frames to sinks sequentially on a serial queue. A blocking sink delays all others.

**Resolution**: All sinks MUST return from `appendVideoFrame:` in < 1ms. Each sink dispatches heavy work internally to its own private queue. Debug assertion enforces this in DEBUG builds.

### Decision 9: Participant Presence [G6]

**Problem**: No "waiting for participant" UI state.

**Resolution**: `InterParticipantPresenceState` enum (`.alone`, `.participantJoined`, `.participantLeft`). 3-second grace period before showing "left" (avoids flicker during reconnections). KVO-driven UI updates.

### Decision 10: Room Discovery [G7]

**Problem**: How does Participant B know which room to join?

**Resolution**: Host-generated 6-char room codes. Token server has `/room/create` and `/room/join` endpoints. Codes expire in 24h. Non-guessable (30^6 = 729M). Redis-backed in production.

### Decision 11: Local Pipeline Isolation [G8]

**Problem**: Network failures must never affect local camera, mic, or recording.

**Resolution**: Architectural invariant — the entire `Networking/` layer is side-effect-free and failure-tolerant. All network errors are swallowed with logging. If `_roomController` is nil, the app functions identically to its current state. CI test enforces this.

### Decision 12: Logging & Diagnostics [G9]

**Problem**: No observability for network events, quality metrics, or debugging.

**Resolution**: `os_log`-based structured logging (zero-cost when not observed). Periodic WebRTC stats collection (10s intervals, 1h circular buffer). On-demand diagnostic snapshot (JSON) for support.

### PF.5 — Interview Mode: Server-Authoritative Room Type & Join Flow Fix

Architecture: **Server-authoritative roles via `roomType`**. The token server is the single source of truth
for whether a room is a "call" or "interview". Joiners never choose their own role — the server
assigns it based on the room type. This is the Zoom/Meet/Teams pattern and is designed for:
- Future multi-interviewer support (each participant has own role in JWT metadata)
- Future dynamic role switching (host calls API → server updates participant metadata → LiveKit broadcasts)
- Future URL-based join (inter://join/CODE, same POST /room/join backend)

- [x] PF.5.1 — Token server: `POST /room/create` accepts `roomType` ("call" | "interview"), stores in room code map, returns in response. Host metadata stamped in JWT (`{"role":"interviewer"}` for interview rooms)

## Phase 4: Production Hardening

### 4.1 — Error Handling Matrix

| Error | Detection | Response | Local Impact |
|-------|-----------|----------|-------------|
| Token server unreachable | InterTokenService timeout | Alert + retry | None |
| Invalid room code [G7] | 404 from /room/join | "Invalid code" alert | None |
| Expired room code [G7] | 410 from /room/join | "Room expired" alert | None |
| Token expired mid-call | RoomDelegate.tokenExpired | Auto-refresh | None |
| WebSocket disconnect | RoomDelegate.didDisconnect | Auto-reconnect. Banner. 30s → manual retry | None |
| ICE failure | didFailToConnect with ICE error | "Check firewall" alert | None |
| Encoder failure | didFailToPublish | Log + "Publishing failed" | None |
| Participant left | participantDidLeave | [G6] Remove views after 3s grace | None |
| Audio bridge error [G1] | Exception in conversion | [G8] Swallowed. os_log | None |
| Screen share error | capturer.capture() fails | [G8] Swallowed. Increment counter | None |
| Memory pressure | DispatchSource memory pressure | Reduce screen share res. Switch to .low simulcast | None |
| SDK crash | WebRTC assertion | [G8] Room disconnects. "Reconnect / Offline" dialog | None |

### 4.2 — Memory Budget

| Component | Max | Enforcement |
|-----------|-----|-------------|
| Camera publish | 30 MB | alwaysDiscardsLateVideoFrames |
| Screen share publish | 60 MB | 1920×1080 cap + 15 FPS throttle |
| Audio bridge [G1] | < 1 MB | 3 × 7.7 KB buffers |
| Remote decode (per track) | 8 MB | Single-slot atomic buffer |
| Remote render (per view) [G3] | 8 MB | 1 CVPixelBuffer + zero-copy textures |
| Stats collector [G9] | < 1 MB | 360 × 100 bytes |
| **Total network overhead** | **< 120 MB** | task_info() monitoring in debug |

### 4.3 — Threading Rules

| Queue | Owner | Rules |
|-------|-------|-------|
| `inter.camera.livekit.encode` | CameraSource | Never blocks. Drops late frames |
| `inter.livekit.audio.bridge` [G1] | AudioBridge | Returns <500ns from router queue |
| `inter.livekit.screenshare.encoder` | ScreenShareSource | Returns <200ns from router queue |
| `secure.inter.media.surface.share.router` | SurfaceShareController | [G5] All sinks <1ms. Debug-asserted |
| LiveKit signaling (internal) | SDK | Don't touch |
| LiveKit WebRTC worker (internal) | SDK | Don't touch |
| CVDisplayLink threads | Surface/Remote views | Never call LiveKit APIs |
| `inter.room.state` | RoomController | All state transitions. Main queue delivery |
| Main thread | AppKit/UI | Never block on network |

Lock discipline: `os_unfair_lock` for frame buffers. `@synchronized` only where already used.

## Addendum — Execution Efficiency Path (No Changes to Original Plan)

> This is a separate implementation track to improve delivery speed and reduce risk.
> It does **not** replace or modify any original phase/step above.

### Goal

Reach first real end-to-end online call capability quickly, then layer full production hardening from the original plan.

### Accelerated Vertical Slice (Recommended Order)

- [ ] A1 — Complete dependency/bootstrap first: **0.3 + 0.2**
  - Add LiveKit SDK, bridging setup, network entitlement.
  - Bring up token server endpoints (`/room/create`, `/room/join`) with secure secret handling.

- [ ] A2 — Build minimum connection control plane
  - Implement minimal `InterRoomController` connect/disconnect.
  - Host/join room-code flow works end-to-end.

- [ ] A3 — Normal mode publish baseline
  - Publish **camera + microphone only** in normal mode.
  - Keep current local pipeline as source of truth.
  - No screen-share publish in this step.

- [ ] A4 — Remote receive baseline
  - Subscribe and render remote camera with a minimal renderer path.
  - Keep this intentionally simple to validate transport reliability first.

- [ ] A5 — Reliability gate before expansion
  - Validate: connect, reconnect, mute/unmute, participant join/leave.
  - Add core logging and basic stats visibility before adding advanced media paths.

- [ ] A6 — Expand to planned advanced architecture
  - Add screen share publish sink.
  - Add custom NV12/BGRA rendering path.
  - Add mode-transition persistence behavior.
  - Add full diagnostics, memory/thread assertions, and complete test matrix.

### Why This Addendum Exists

- Reduces first-value time and integration uncertainty.
- Prevents large-batch failures from 2.7k+ lines of new networking/media code landing at once.
- Preserves the original production-grade architecture while improving execution efficiency.
